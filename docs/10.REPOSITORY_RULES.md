# REPOSITORY_RULES.md

Repository Structure and File Management Rules for AI-Assisted
Development

This document defines **strict repository rules** that AI agents must
follow when creating, editing, or organizing files within this project.

The purpose of this document is to prevent:

-   project structure corruption
-   duplicate files
-   inconsistent naming
-   accidental overwrites
-   architecture drift

AI must read this document before performing repository-level
operations.

Related documents: - 00.PRD.md - 01.PROJECT_BRIEF_FTP.md or
01.PROJECT_BRIEF_CLOUDFLARE.md - 02.PROJECT_CONTEXT.md -
03.DESIGN_SYSTEM.md - 04.SEO_RULES.md - 05.COMPONENT_LIBRARY.md -
07.AI_EXECUTION_RULES.md - 08.PAGE_TEMPLATES.md

------------------------------------------------------------------------

# 1. Core Repository Principles

The repository must remain:

-   predictable
-   organized
-   scalable
-   AI-readable

AI must prioritize **stability over novelty**.

Never restructure the repository unless explicitly instructed.

------------------------------------------------------------------------

# 2. Approved Directory Structure

The default project structure is:

project-root/

00.PRD.md 01.PROJECT_BRIEF.md 02.PROJECT_CONTEXT.md 03.DESIGN_SYSTEM.md

/public /pages /assets /assets/css /assets/js /assets/images
/assets/icons /includes /blog

------------------------------------------------------------------------

# 3. File Naming Rules

File names must follow these principles:

-   lowercase preferred for web files
-   descriptive names
-   avoid spaces
-   use hyphens when needed

Examples:

good: about.html product-detail.html main-navigation.js

bad: AboutPage.HTML my file.html scriptFINAL.js

------------------------------------------------------------------------

# 4. File Creation Rules

AI may create files only when:

1.  the feature clearly requires a new file
2.  no existing file already handles that responsibility
3.  the file fits into the defined directory structure

Before creating a file, AI must check:

-   whether a similar file already exists
-   whether a reusable component already exists

Avoid duplicate functionality.

------------------------------------------------------------------------

# 5. File Modification Rules

AI must prefer **editing existing files** over creating new ones.

AI must not:

-   rename core documentation files
-   delete project documents
-   overwrite configuration files
-   remove SEO-critical HTML structures

Protected files include:

00.PRD.md PROJECT_BRIEF files PROJECT_CONTEXT.md DESIGN_SYSTEM.md
SEO_RULES.md COMPONENT_LIBRARY.md

These files should only change when explicitly instructed.

------------------------------------------------------------------------

# 6. Asset Management Rules

All assets must be organized correctly.

CSS: /assets/css/

JavaScript: /assets/js/

Images: /assets/images/

Icons: /assets/icons/

Rules:

-   never place images directly in root
-   never mix CSS and JS directories
-   avoid large uncompressed images

------------------------------------------------------------------------

# 7. Public Page Rules

Public-facing pages must remain:

-   lightweight
-   SEO-friendly
-   semantic

Public pages must be implemented with:

HTML CSS minimal JavaScript

AI must avoid introducing:

-   large frameworks
-   unnecessary dependencies
-   heavy client-side logic

------------------------------------------------------------------------

# 8. Application Layer Rules

Authentication features belong to the **application layer**, not the
public site.

Examples:

-   login
-   signup
-   admin dashboard
-   user management

These may use frameworks such as:

Next.js React

But public pages should remain framework-light.

------------------------------------------------------------------------

# 9. Component Consistency Rules

Before creating UI markup, AI must check:

COMPONENT_LIBRARY.md

Rules:

-   reuse components
-   extend components instead of duplicating
-   maintain spacing and style tokens

Never create inconsistent UI structures.

------------------------------------------------------------------------

# 10. Safe Refactoring

When improving the codebase:

AI must:

1.  maintain compatibility
2.  avoid breaking existing pages
3.  preserve SEO structures
4.  maintain component contracts

Large structural refactors require explicit approval.

------------------------------------------------------------------------

# 11. Commit-Level Thinking

Even when not using Git directly, AI should behave as if each change is
a commit.

Rules:

-   make small focused changes
-   avoid massive rewrites
-   keep modifications understandable

------------------------------------------------------------------------

# 12. AI Execution Protocol

Before performing any repository operation, AI must:

1.  read the documentation set
2.  understand project constraints
3.  verify file structure
4.  determine the minimal safe change

Then execute the task.

------------------------------------------------------------------------

# End of Repository Rules
