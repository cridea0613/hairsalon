# AI_EXECUTION_RULES.md

AI Safety and Execution Rules for Vibe Coding Projects

This document defines **strict operational rules** for AI agents working
on this project. Its purpose is to prevent AI from breaking the project
structure, overwriting important files, or introducing architectural
inconsistencies.

AI must read this document **before modifying or generating any code**.

Related documents: - 00.PRD.md - 01.PROJECT_BRIEF_FTP.md or
01.PROJECT_BRIEF_CLOUDFLARE.md - 02.PROJECT_CONTEXT.md -
03.DESIGN_SYSTEM.md - 04.SEO_RULES.md - 05.COMPONENT_LIBRARY.md -
06.PROMPT_LIBRARY.md

------------------------------------------------------------------------

# 1. General AI Behavior Rules

AI must:

1.  Read all relevant project documents before generating code.
2.  Respect the existing folder structure.
3.  Follow the design system and component library.
4.  Avoid unnecessary frameworks for public-facing pages.
5.  Keep solutions simple and maintainable.

AI must NOT:

-   rewrite large portions of the project without instruction
-   introduce heavy frameworks into static public pages
-   duplicate existing components
-   modify core architecture without approval

------------------------------------------------------------------------

# 2. File Modification Rules

Before editing any file, AI must:

1.  Identify whether the file already exists.
2.  Check whether a component already solves the problem.
3.  Prefer **modifying existing code** rather than creating duplicates.

AI must NOT:

-   delete files without explicit instruction
-   rename critical project documents
-   overwrite configuration files

Protected files:

-   00.PRD.md
-   PROJECT_BRIEF files
-   PROJECT_CONTEXT.md
-   DESIGN_SYSTEM.md
-   SEO_RULES.md

These files should **never be modified automatically**.

------------------------------------------------------------------------

# 3. Component Usage Rules

AI must always check `COMPONENT_LIBRARY.md` before creating UI.

Rules:

-   reuse existing components
-   extend components when necessary
-   avoid creating new UI patterns unnecessarily

If a new component is needed:

1.  verify that no existing component fits
2.  follow naming conventions
3.  maintain spacing and design tokens

------------------------------------------------------------------------

# 4. Public Website Rules

Public-facing pages must prioritize:

-   SEO
-   accessibility
-   fast loading
-   semantic HTML

Public pages must be built with:

-   HTML5
-   CSS
-   minimal JavaScript

AI must avoid:

-   large JS frameworks
-   unnecessary client-side rendering
-   heavy animations

------------------------------------------------------------------------

# 5. Application Layer Rules

Protected features such as:

-   login
-   signup
-   admin dashboard
-   user management

belong to the **application layer**, not the public site.

These may use:

-   Next.js
-   React
-   API integrations

AI must clearly separate:

Public SEO site and Authenticated application layer

------------------------------------------------------------------------

# 6. Code Quality Rules

AI-generated code must:

-   be readable
-   follow consistent naming
-   avoid duplicate styles
-   avoid inline CSS where possible
-   keep JavaScript modular

Preferred naming style:

.hero-section .nav-container .feature-card .cta-button

------------------------------------------------------------------------

# 7. SEO Protection Rules

AI must preserve SEO-critical elements:

-   page titles
-   meta descriptions
-   heading hierarchy
-   structured data
-   canonical tags

AI must NOT:

-   remove SEO metadata
-   break heading hierarchy
-   replace semantic HTML with generic div structures

------------------------------------------------------------------------

# 8. Performance Rules

AI must prioritize performance.

Guidelines:

-   minimize JavaScript usage
-   lazy-load images
-   optimize image formats (WebP)
-   avoid blocking scripts
-   keep CSS modular

Target Lighthouse scores:

Performance ≥ 90 Accessibility ≥ 95 SEO ≥ 95

------------------------------------------------------------------------

# 9. Safe Refactoring Rules

When improving existing code:

AI must:

1.  preserve functionality
2.  maintain design system rules
3.  keep component compatibility
4.  avoid large breaking changes

If a major change is required:

AI should recommend the change instead of automatically applying it.

------------------------------------------------------------------------

# 10. Execution Protocol

Before executing a development task, AI must:

1.  Read project documentation in order:

    -   00.PRD.md
    -   PROJECT_BRIEF
    -   PROJECT_CONTEXT
    -   DESIGN_SYSTEM
    -   SEO_RULES
    -   COMPONENT_LIBRARY

2.  Understand constraints.

3.  Generate the smallest safe change required.

AI must prioritize **stability over novelty**.

------------------------------------------------------------------------

# End of AI Execution Rules
